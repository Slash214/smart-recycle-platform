# 管理端订单状态与逻辑重构开发文档

## 1. 目标与问题

当前管理端订单模块主要问题：

- 状态维度过粗，仅有 `status(1/2/3)`，无法表达“待报价/已报价/退货中”等真实业务节点。
- 表单与列表字段不统一：`way` 在部分页面仍按文本输入，已与后端 `enum(1,2,3)` 冲突。
- 小程序与管理端状态口径不同，导致联调成本高、问题反复。

本次重构目标：管理端与小程序共享同一套状态模型，形成可运营、可筛选、可追踪的订单状态体系。

---

## 2. 统一业务状态（管理端口径）

## 2.1 状态字段建议

- `inboundStatus`（入库状态）
  - `10` 待入库
  - `20` 已入库
- `settlementStatus`（结算状态）
  - `10` 待报价
  - `20` 已报价
  - `30` 待结算
  - `40` 已结算
  - `50` 退货中

保留旧字段：

- `status`：兼容历史数据/旧接口使用，逐步废弃。

## 2.2 收款方式统一

- `way`（收款方式）
  - `1` 微信收款
  - `2` 支付宝收款
  - `3` 银行卡收款

---

## 3. 管理端页面重构范围

涉及页面：

- `admin/src/pages/orders/list.tsx`
- `admin/src/pages/orders/create.tsx`
- `admin/src/pages/orders/edit.tsx`
- `admin/src/pages/orders/show.tsx`

## 3.1 列表页（List）

筛选区建议增加：

- 关键词（手机号/单号/订单号）
- 入库状态（单选）
- 结算状态（单选）
- 类型 `type`（上门/邮寄）
- 收款方式 `way`（微信/支付宝/银行卡）
- 时间区间（startAt/endAt）

列表列建议：

- 订单号
- 用户手机号
- 数量
- 类型
- 收款方式（Tag）
- 快递公司 / 运单号
- 入库状态（Tag）
- 结算状态（Tag）
- 创建时间
- 操作（查看/编辑/删除）

## 3.2 创建/编辑页（Create/Edit）

字段规则：

- `way` 必须使用 `Select`，禁止 `Input`
- `type` 使用 `Select(1/2)`
- `inboundStatus`、`settlementStatus` 使用 `Select`
- `remark_images` 使用 URL 数组输入组件

提交前转换：

- 强制 number 化：`nums/type/way/inboundStatus/settlementStatus`
- 空数组默认：`remark_images=[]`

## 3.3 详情页（Show）

- 同时展示“入库状态 + 结算状态”
- `way` 显示映射文案，不直接显示数字
- `remark_images` 支持预览组

---

## 4. 数据接口与参数契约（管理端）

## 4.1 列表查询参数（建议）

- `GET /api/v1/orders`
- Query：
  - `page`
  - `pageSize`
  - `keyword`
  - `type`
  - `way`
  - `inboundStatus`
  - `settlementStatus`
  - `startAt`
  - `endAt`

## 4.2 创建/更新请求体（建议）

```json
{
  "nums": 12,
  "phone": "153289842222",
  "type": 2,
  "way": 1,
  "tracking_number": "SF123456789",
  "express_company": "顺丰快递",
  "remark": "备注",
  "remark_images": [
    "https://img.example.com/a.jpg"
  ],
  "inboundStatus": 10,
  "settlementStatus": 10
}
```

---

## 5. 与后端联动改造点（给管理端开发关注）

- 后端 `schema` 增加并校验：
  - `way: integer enum[1,2,3]`
  - `inboundStatus`
  - `settlementStatus`
- 后端列表查询支持新筛选参数。
- 后端返回统一对象，前端无需猜字段（避免 `shippingMethod/courier` 这类旧字段）。
- 兼容期允许旧 `status`，但管理端 UI 只使用新状态。

---

## 6. 任务拆分（管理端）

- A1：状态常量与映射工具封装（Tag、Label、Select options）
- A2：列表页筛选区与列重构
- A3：创建页与编辑页 `way` 输入改为枚举
- A4：详情页状态区重构
- A5：DataProvider 参数透传与联调验证

---

## 7. 验收标准（管理端）

- 列表可按入库状态、结算状态、关键词组合筛选。
- 创建/编辑不再出现 `way must be integer`、`remark_images` 类型报错。
- 列表、详情、编辑页状态文案一致。
- 管理端与小程序同一订单状态展示一致。

---

## 8. 迁移策略（建议）

- 第 1 阶段：后端新增字段并兼容旧 `status`。
- 第 2 阶段：管理端和小程序全部切新字段，保留旧字段只读。
- 第 3 阶段：移除旧状态映射与历史分支逻辑。

为避免线上风险，建议增加一份状态变更审计（记录变更前后状态、操作者、时间）。
