# 小程序端订单状态与逻辑重构开发文档

## 1. 目标与背景

当前小程序订单模块存在以下问题：

- 状态定义不统一：`my` 页显示“待确认/进行中/已完成”，`order-list` 页显示“已发货/已收货”，与接口文档和管理端不一致。
- 字段语义混乱：`type`、`way`、`shippingMethod`、`courier` 在页面与接口中的含义冲突。
- 筛选结构无法支撑业务：现有仅 `status(1/2/3)`，无法覆盖“入库流程 + 结算流程”的组合状态。

本次目标是将小程序订单页重构为**双维度状态模型**，并统一接口参数与页面结构，满足截图里的业务交互：

- 一级：`入库订单`、`结算状态`
- 二级（示例）：
  - 入库订单：`待入库单`、`待结算单`
  - 结算状态：`待报价`、`已报价`、`待结算`、`已结算`、`退货中`

---

## 2. 统一状态模型（前端约定）

> 小程序前端不再直接依赖旧的 `status(1/2/3)` 文案映射，而是基于新字段渲染。

### 2.1 流程维度

- `inboundStatus`（入库状态）
  - `10` 待入库
  - `20` 已入库
- `settlementStatus`（结算状态）
  - `10` 待报价
  - `20` 已报价
  - `30` 待结算
  - `40` 已结算
  - `50` 退货中

### 2.2 页面过滤维度

- 一级标签（主视图）
  - `INBOUND`（入库订单）
  - `SETTLEMENT`（结算状态）
- 二级标签（子视图）
  - 当主视图为 `INBOUND`：
    - `PENDING_INBOUND` -> `inboundStatus=10`
    - `PENDING_SETTLEMENT` -> `inboundStatus=20 && settlementStatus in (10,20,30)`
  - 当主视图为 `SETTLEMENT`：
    - `WAIT_QUOTE` -> `settlementStatus=10`
    - `QUOTED` -> `settlementStatus=20`
    - `WAIT_SETTLE` -> `settlementStatus=30`
    - `SETTLED` -> `settlementStatus=40`
    - `RETURNING` -> `settlementStatus=50`

---

## 3. 页面结构重构（小程序）

## 3.1 页面信息架构

- 页面：`/pages/order-list/order-list`
- 顶部：
  - 返回按钮
  - 标题“我的订单”
  - 一级标签（入库订单 / 结算状态）
- 中部：
  - 关键词搜索框（手机号/单号/订单号）
  - 二级状态标签（随一级标签动态变化）
- 列表区：
  - 卡片列表（支持分页）
  - 空态
- 底部：
  - 固定按钮“我要退货”（仅在结算态下显示）

## 3.2 交互规则

- 切换一级标签：
  - 重置二级标签为该一级下默认项
  - 清空分页并重新请求
- 切换二级标签：
  - 重置分页并重新请求
- 输入关键词点击搜索：
  - `keyword` 入参传给接口
- 上拉加载：
  - `page += 1` 并 append 列表
- 下拉刷新：
  - 重置页码，重新拉取

---

## 4. 接口参数与返回（小程序端消费契约）

## 4.1 列表接口（建议）

- `GET /api/v1/orders`
- Query：
  - `page` number
  - `pageSize` number
  - `keyword` string（可选）
  - `inboundStatus` number（可选）
  - `settlementStatus` number（可选）

> 如果后端暂未支持新参数，可先沿用 `status`，但必须提供映射层（见 4.3）。

## 4.2 建议返回字段

- `id`
- `orderNo`
- `phone`
- `nums`
- `type`
- `way`
- `tracking_number`
- `express_company`
- `remark`
- `remark_images`
- `inboundStatus`
- `settlementStatus`
- `createdAt`
- `updatedAt`

## 4.3 兼容期映射策略（临时）

当后端仅返回 `status(1/2/3)` 时，小程序通过适配函数转换：

- `status=1` -> `inboundStatus=10, settlementStatus=10`
- `status=2` -> `inboundStatus=20, settlementStatus=30`
- `status=3` -> `inboundStatus=20, settlementStatus=40`

> 该映射仅用于过渡，后续应移除并切换到真实字段。

## 5. 前端代码重构建议（目录级）

## 5.1 常量统一

新增 `app/src/constant/order.ts`：

- `ORDER_MAIN_TAB`
- `INBOUND_SUB_TAB`
- `SETTLEMENT_SUB_TAB`
- `INBOUND_STATUS_LABEL`
- `SETTLEMENT_STATUS_LABEL`
- `PAY_WAY_LABEL`

## 5.2 类型统一

新增/更新 `app/src/models/order.ts`：

- `OrderDTO`（接口原始）
- `OrderViewModel`（页面展示）
- `OrderListQuery`

## 5.3 API 层

更新 `app/src/apis/index.ts`：

- `getOrderList(params)` 入参改为新筛选结构
- 创建 `mapOrderListQuery()`，只传有效参数（避免 `undefined`）

## 5.4 页面与组件

- `order-list.vue`：
  - 拆分状态 tab 配置
  - 抽离 `buildQueryByTab()`
  - 分离 `fetchFirstPage()` / `fetchNextPage()`
- `OrderCard.vue`：
  - 字段改为真实接口字段名（`tracking_number` 等）
  - 状态展示改为 `settlementStatus` 文案
  - 底部按钮区域按状态控制（如“确认收货”“申请退货”）

## 7. 验收标准（小程序）

- 切换一级/二级标签，列表结果与状态一致。
- 不再出现 `status=undefined`、类型错误参数。
- “入库订单 / 结算状态”二层结构与设计稿一致。
- `remark_images`、`way`、`type` 字段展示稳定。
- 搜索、分页、空态全部正常。

---

## 8. 风险与注意事项

- 后端若未提供 `inboundStatus`/`settlementStatus`，前端必须做兼容映射且加 TODO 标记。
- 同步统一 `way` 含义：约定为收款方式 `1微信 2支付宝 3银行卡`，不要再混用“物流方式”文案。
- 旧组件字段（`shippingMethod`、`senderPhone`）需逐步替换，避免脏字段继续扩散。
